# Product Strategy

도메인 인사이트를 프로덕트 청사진으로 변환하는 스킬. spec-writer가 XML 스펙을 쓸 수 있도록 "무엇을 만들 것인가"를 결정한다.

## 워크플로우

### 1. 입력 확인

- `_workspace/01_domain_insights.md` 읽기 (필수)
- 사용자 원본 요청 재확인
- 기존 `_workspace/02_product_blueprint.md` 존재 시 갱신 모드

### 2. 청사진 섹션별 작성

#### 2-1. 비전 / 가치 제안
- 한 줄 요약 (Elevator Pitch)
- 차별화 포인트 3개 이내

예: "초·중학교 생활기록부에서 학생의 강점·관심사를 자동 분석하여, 고1~고3 단계별 맞춤 TODO 로드맵을 제공하는 학생용 진학 코치 서비스. 일반 진학 정보 사이트와 달리 *내 생기부에 기반한* 개인화 가이드를 즉시 받을 수 있다."

#### 2-2. 페르소나 (2~3개)
각 페르소나에 대해:
- 이름·나이·배경 (구체적으로)
- 주요 목표 (Goal)
- 페인 포인트 (Pain Point)
- 기술 친숙도
- 사용 맥락 (언제, 어디서, 어떤 기기로)

권장 페르소나:
- **A. 예비 고1 학생** (가장 본질적 사용자)
- **B. 학부모** (실 결제자, 자녀와 함께 사용)
- **C. (선택) 진학 상담 교사** — 전문가 사용자 트랙 (v2+)

#### 2-3. 핵심 기능 (MVP)

테이블 형식:

| 기능 | 설명 | 우선순위 | 페르소나 매핑 |
|-----|-----|--------|------------|
| 생기부 PDF 업로드 | 드래그앤드롭, 다중 파일 | P0 | A, B |
| 자동 분석 | LLM으로 학생 프로필 추출 | P0 | A, B |
| 트랙 선택 | 정시/수시 트랙 선택 → TODO 분기 | P0 | A, B |
| 학년별 TODO 보드 | 고1/고2/고3 카드 보드 | P0 | A |
| 카테고리 필터 | 학습·진로·활동·시험·자기관리 | P0 | A |
| 진행 추적 | 체크박스, 완료율 | P0 | A, B |
| 학년 전환 | 고1 → 고2 자동 갱신 | P1 | A |
| ... | | | |

P0 = MVP 필수 / P1 = 출시 후 1차 / P2 = v2+

#### 2-4. 확장 기능 (v2+)

- 학교별 TODO 커스터마이징
- 또래 비교 (익명 통계)
- 멘토/상담사 연결
- 학부모 대시보드
- 학원/온라인강의 추천 연동
- 푸시 알림 / 캘린더 동기화
- 자기소개서 초안 생성 (고3 후반)

#### 2-5. 핵심 사용자 흐름

각 흐름에 대해 단계별 액션 + 화면 + 결과를 명시:

**Flow 1: 첫 진입 → TODO 생성 (Happy Path)**
1. 랜딩 → 가입 → 동의 (개인정보, 생기부 처리)
2. 생기부 PDF 업로드
3. 분석 진행 (로딩, 예상 시간 표시)
4. 분석 결과 요약 (강점·관심사·약점)
5. 입시 트랙 선택 (또는 추천)
6. 학년별 TODO 보드 진입
7. 첫 TODO 항목 확인

**Flow 2: TODO 진행 추적 (재방문)**
**Flow 3: 학년 전환 (고1 → 고2)**
**Flow 4: PDF 재업로드 / 갱신**
**Flow 5: 개인정보 삭제 요청**

각 Flow마다 엣지 케이스 1~2개 (예: PDF 인식 실패, 분석 시간 초과).

#### 2-6. 화면 구성 (개념)

상세 픽셀/색상은 spec-writer가 결정. 여기서는 화면 목록과 핵심 요소만:

- 랜딩 페이지 / 회원가입·로그인 / 동의서 화면
- 대시보드 (분석 결과 요약 + 학년 진행 상태)
- 업로드 화면 / 분석 진행 화면
- 학년 TODO 보드 (고1/고2/고3 탭)
- TODO 상세 화면 (설명, 자료, 완료 처리)
- 트랙 선택 / 변경 화면
- 마이페이지 / 데이터 관리 (다운로드/삭제)
- 설정 (알림, 학년 전환 시점)

#### 2-7. 개념 데이터 모델

엔티티 + 관계만 (필드 디테일은 spec-writer):

- **User** (학생/학부모)
- **StudentProfile** (분석된 학생 프로필)
- **Record** (업로드된 생기부 메타)
- **AnalysisResult** (LLM 분석 결과)
- **Track** (정시/수시 등)
- **TODOItem** (학년·카테고리·트랙별)
- **TODOTemplate** (도메인 규칙 기반 마스터 템플릿)
- **Progress** (사용자별 TODO 진행)
- **ConsentLog** (동의 이력)

관계: User 1:N Record, Record 1:1 AnalysisResult, AnalysisResult N:M TODOItem (개인화 후 생성), User 1:N Progress 등

#### 2-8. 차별화 포인트

- 진학사·메가스터디 등 대비 *내 생기부 기반* 개인화
- AI 분석 → 일반론이 아닌 맞춤 조언
- 학년 전환 자동 갱신 → 한 번 가입으로 3년 사용
- 민감정보 즉시 삭제 옵션 → 신뢰

#### 2-9. KPI

- **활성도**: 가입 후 7일 내 PDF 업로드율
- **가치 인지**: 분석 결과 페이지 평균 체류 시간
- **재방문**: WAU/MAU
- **TODO 완료**: 월간 TODO 완료율 (학년별)
- **유료 전환** (v2+): 학년 전환 시 갱신 결제율

#### 2-10. 개인정보 / 보안 요구사항 (개념)

- 생기부는 민감정보 → 명시적 동의
- PDF 원본 저장 옵션 (저장 / 분석 후 즉시 삭제 / 30일 보관)
- 분석 결과는 사용자 계정에 저장 (암호화)
- LLM 호출 시 PII 마스킹 (이름, 학교명 등)
- 사용자 삭제 요청 시 24시간 내 완전 삭제
- 만 14세 미만은 법정대리인 동의

### 3. spec-writer를 위한 메모

청사진 끝부분에 spec-writer가 참고할 메모 작성:
- 권장 기술 스택 (선택 사항)
- 절대 깨지면 안 될 제약 (CRITICAL)
- 의도적으로 보류한 결정 (spec-writer가 채울 부분)

### 4. 산출물 저장

`_workspace/02_product_blueprint.md`로 저장.

## 작성 원칙

- **사용자 중심**: 모든 기능은 페르소나 페인 포인트와 연결.
- **MVP 우선**: P0/P1/P2 명확히 구분.
- **현실적 범위**: 3~6개월 내 소규모 팀이 빌드 가능 수준.
- **개인정보 의식**: 생기부는 민감정보. 동의·저장·삭제 정책을 처음부터 포함.
- **구체 페르소나**: 이름·나이·학교 유형·기기 등 디테일이 있어야 spec-writer가 UI 결정 가능.

## 후속 작업 모드

- **신규**: 전체 워크플로우
- **기능 추가**: 해당 섹션만 보강 (페르소나/Flow/데이터 모델 영향 일관 갱신)
- **페르소나 변경**: 페르소나 → Flow → 기능 → KPI까지 전파 갱신
